# TenniSync — 网球 3D 追踪与分析系统

## 项目目标

用两个固定 IP 摄像头实时追踪网球比赛中的球，输出：
1. 球的 3D 实时轨迹
2. 落点判定（IN/OUT）
3. 球速（过网速度）
4. 回合分割与比赛统计
5. 3D 大屏可视化

```

                                                         │
                                                    top1_conf20
                                                    (去掉最弱 20%)
                                                         │
                                              Homography → 世界坐标
                                                         │

                                              ┌──────────┼──────────┐
                                              │          │          │
                                         视频渲染    3D大屏WS    JSON统计
```

## 核心算法选型（经 GT 验证）

### 球检测: TrackNet V3 变体
- 8 帧输入，输出 heatmap
- Raw recall: 99.4%（473 帧 GT 中检测到 468 帧）
- 不依赖球的外观特征，利用多帧间画面差异检测运动目标
- **结论: 检测不是瓶颈，无需换模型**

### 球跟踪/匹配: top1_conf20（置信度过滤）
经 15 种方法系统对比，最简单的方法最优：

| 方法 | Recall | Precision | F1 | <10px | Jitter |
|------|:---:|:---:|:---:|:---:|:---:|
| **top1_conf20** | **91.5%** | **39.2%** | **54.9%** | **95.8%** | **18.0** |
| MultiBlobMatcher | 79.5% | 35.0% | 48.6% | 95.5% | 23.2 |
| Viterbi 全局优化 | 75.1% | 33.6% | 46.5% | 95.8% | 31.5 |
| 跨相机验证 | 82.2% | 37.9% | 51.9% | 95.4% | 17.4 |

**核心发现: 简单的 heatmap 置信度过滤 > 复杂的跨相机匹配/全局优化**
- Viterbi 全局最优听起来好，但实际上更抖（blob 切换更频繁）
- MultiBlobMatcher 的跨相机约束太强，丢帧多
- TrackNet 的 blob_sum 本身就是很好的信号：真球响应强，干扰物响应弱

### 回合分割: conf_density 或 conf_gap
| 方法 | Recall | Precision | F1 |
|------|:---:|:---:|:---:|
| conf_gap_g25_s30 | 99.2% | 37.9% | 54.8% |
| conf_density_w40_m15 | 96.4% | 38.4% | 54.9% |

利用高置信度检测的密度/间隔来识别回合——回合期间检测密集，间歇期间稀疏。

### Bounce 检测: V-shape（Z 轴局部极小值）
- 两轮检测: strict (z<0.5m) + relaxed (z<0.8m)
- 非对称 V-shape 支持快速回球
- Court bounds + dense segment 过滤
- 当前: P=84.6%, R=73.3%, F1=78.6%
- **精度瓶颈不在算法，在上游数据质量**（相机同步、Z 轴精度）

### IN/OUT 判定: 单相机 Homography
- Bounce 瞬间球在地面（z≈0），homography 地面假设成立
- 球在哪半场就用哪个相机的 homography（近端 cam66，远端 cam68）
- 理论精度 ~5cm（homography 重投影误差）
- 比三角化 3D 坐标（1.5m 误差）准确 30 倍

## 已发现的关键问题

### 1. 标定坐标系错误 ⚠️
- 标注点是单打线（x=1.37, 6.86），但代码映射到双打线（x=0, 8.23）
- 所有 x 坐标被拉伸 1.5 倍
- 影响: homography、三角化、IN/OUT 判定全部偏移
- 状态: **已修正 compute_homography.py 和 calibration.py**

### 2. 相机帧不同步
- 两相机帧偏移 1-2 帧（40-80ms）
- 球速 30m/s 时 = 1.2-2.4m 3D 位移误差
- 是远端 Z 精度差的主要原因（不是 homography 的问题）
- NVR 无法解决（只同步时钟，不同步快门）
- **最佳方案: 音频互相关帧对齐**（利用击球声，精度 <1ms）

### 3. Precision 天花板 ~40%
- 非比赛时段场上也有球（废球、放置的球、练习球）
- TrackNet 正确检测到了这些球，但它们不是 match_ball
- **只能通过回合分割解决**（回合外不输出检测结果）

## 合成数据（Unity）

### 已搭建
- 标准网球场 + 两个虚拟相机（匹配真实位置）
- 两个简化球员（primitive 人形）
- 完整的对打模拟（发球、topspin/flat/slice/dropshot/winner）
- 球物理: 重力 + 空气阻力 + Magnus 效应 + 旋转弹跳
- 人体极限: 反应时间、加速度、最大速度、击球高度随机化
- 技能等级随机（0.3-0.85）

### 已生成数据
- 500,000 帧 / 19 分钟
- 91 个回合, 527 次击球, 558 次弹跳
- 含事件标签（serve/hit/bounce/fly）+ rally_id

### 用途
- 训练 3D 轨迹事件分类器（bounce/shot/serve）
- 不用于训练视觉模型（domain gap 问题）
- 需要噪声注入匹配真实数据的误差分布

## 3D 大屏可视化

- WebSocket: wss://tennisserver.motionrivalry.com:8086/general
- 格式: bounce_data 消息（x, y, speed）
- 坐标归一化: x=[0,1] 单打宽度，y=[0,1] 球场长度
- 大屏是双打球场显示 → 需要转换: display_x = 0.1665 + x * 0.6671
